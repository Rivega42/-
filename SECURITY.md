# Security Policy

## Поддерживаемые версии

BookCabinet — проект в active development. Поддерживается только текущая ветка `main`.

| Версия | Поддержка |
|---|---|
| `main` (latest) | ✅ |
| Старые коммиты | ❌ — обновляйтесь до `main` |

## Сообщить об уязвимости

Если вы обнаружили security уязвимость, **не открывайте публичный issue**.

### Предпочтительный путь
- **GitHub Security Advisories** — https://github.com/Rivega42/-/security/advisories/new
  (Private vulnerability reporting должен быть включён в репо — см. issue для Vika в #META.)

### Альтернативный путь
- Связаться с владельцем репо приватно через GitHub (`@Rivega42`).
- Тема сообщения: `[SECURITY] BookCabinet — короткое описание`.

### Что включить в отчёт
- Описание уязвимости и потенциальный impact.
- Шаги для воспроизведения (PoC если возможно).
- Затронутые файлы / endpoints / GPIO операции.
- Версия / коммит, на котором обнаружено.
- Предложения по mitigation (опционально).

## Время реакции

- **Подтверждение получения отчёта:** ≤ 72 часов
- **Первичная оценка severity:** ≤ 7 дней
- **Patch для критических уязвимостей:** ≤ 14 дней (если возможно)

## Threat model (краткая)

BookCabinet работает на Raspberry Pi внутри публичной библиотеки. Основные угрозы:

| Категория | Пример | Mitigation |
|---|---|---|
| Физический доступ | злоумышленник за корпусом шкафа | механическая защита, замки, видеонаблюдение библиотеки |
| Сетевой доступ | кто-то на той же сети | локальная сеть, нет внешних портов наружу |
| ИРБИС integration | компрометация креденшелов | креденшелы в `.env` (не в репо), ротация |
| RFID spoofing | подделка ЕКП | ИРБИС-валидация ID, не доверять только UID |
| Telegram bot | leak токена | `.env`, не коммитить, ротация при подозрении |
| Supply chain | вредоносный npm/pip пакет | `package-lock.json`, `uv.lock`, Dependabot, secret scanning |

## Best practices для контрибьюторов

- Никогда не коммитьте `.env`, `*.key`, `*.pem`, токены, креденшелы.
- Используйте `.env.example` как шаблон. Реальные значения — только локально.
- Перед PR проверьте diff на наличие секретов.
- При работе с GPIO — соблюдайте safety rules из `CLAUDE.md` §3.
- При обнаружении секрета в истории — НЕМЕДЛЕННО создайте P0 issue для Vika и не закрывайте его до ротации ключа.
