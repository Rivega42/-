## 🏗️ Архитектурные решения (требуют обсуждения)

### [ ] Два параллельных пути выдачи/возврата книг (TS vs Python)
- **Проблема:** TypeScript (`routes.ts` + `cabinetService.ts`) и Python (`bookcabinet/business/issue.py`) реализуют одну и ту же логику выдачи/возврата по-разному
- **TS путь:** вызывается из React UI → обновляет БД → вызывает ИРБИС. Механическая верификация — симуляция (hardcoded `book_sensor = true`)
- **Python путь:** полная механическая верификация через `algorithms.take_shelf()` → реальные сенсоры → потом БД → потом ИРБИС
- **Риск:** Если UI использует TS-путь — книга помечается выданной без физической проверки
- **Варианты решения:**
  1. TS вызывает Python через subprocess/HTTP — единый путь через Python
  2. Удалить Python бизнес-логику, перенести всё в TS (теряем реальную механику)
  3. TS → Python через WebSocket/RPC (добавляет сложность)
- **Рекомендация:** Вариант 1 — TS как UI-слой, Python как единственный исполнитель бизнес-логики

### [ ] Два дублирующих UI (React + Python static)
- React UI (`client/`) — основной, с калибровкой
- Python static UI (`bookcabinet/server/static/`) — имеет Teach Mode и Settings, которых нет в React
- **Рекомендация:** Перенести Teach Mode и Settings в React, удалить Python UI
