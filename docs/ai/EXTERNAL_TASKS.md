# External Tasks — задачи для внешних исполнителей

> Когда нужно подключить человека/специалиста извне (mechanical engineer, дизайнер, аудитор), задача формализуется здесь.

---

## Когда нужен external

- Физическая работа с шкафом в библиотеке (Roman или его прокси).
- Mechanical / электрика — ремонт, замена компонентов.
- Дизайнер — UX research / прототипы.
- Аудитор — security audit перед production.
- Юрист — лицензия, ToS, обработка ПД.

## Формат external task

```markdown
Title: [external:<тип>] <короткое описание>
Labels: external, тип-исполнителя, P0|P1|P2|P3

## Кому
Имя / роль исполнителя. Контакт (если внутри проекта).

## Цель
Что должно произойти и зачем.

## Контекст
Что уже сделано / зафиксировано / попробовано.

## Что от исполнителя нужно
Список конкретных шагов / поставок.

## Риски / safety
Особенно для физической работы.

## Acceptance
- [ ] критерий 1
- [ ] критерий 2

## Deadline
YYYY-MM-DD или "до production deploy" / "до конца Q2" / etc.

## Бюджет (если применимо)
сумма / часы.

## Связанные
ссылки на ROADMAP, issues, фото/видео.
```

## Текущие категории external

### `external:hardware`
- Mechanical / motor / belt issues.
- Замена компонентов.
- Калибровка на месте.
- Roman или mechanical engineer.

### `external:design`
- UX research, прототипы, иллюстрации.
- Иконки, шрифты, brand assets.

### `external:legal`
- Лицензия (если выбираем не MIT).
- ToS для пользователей.
- Соглашение с библиотекой.
- Обработка ПД (152-ФЗ).

### `external:audit`
- Security audit перед production.
- Code audit от стороннего эксперта.
- Penetration test (если будет внешний доступ).

### `external:translation`
- RU → EN перевод UI и docs (Q4 2026).

## Безопасность external'ов

- Не давать write-доступ к main репо. Forks + PR.
- Секреты — никогда не передавать external'ам.
- Hardware работа — фиксировать до/после фотографиями.
- После завершения — ротация любых ключей, к которым был доступ.
