# Правила дизайна UI BookCabinet

> Применимо к kiosk UI и любым операторским интерфейсам.

---

## Контекст использования

- **Kiosk UI** — публичный шкаф в библиотеке. Touch-screen.
- **Операторский UI** — браузер на ноуте библиотекаря (опционально).
- **Пользователи:** читатели библиотеки, разный возраст, разная техническая грамотность, иногда — пожилые.
- **Освещение:** разное в течение дня. Может быть яркое естественное.
- **Время на действие:** короткое (читатель не должен застревать).

---

## Палитра и контраст

- **ЧБ-первый подход.** Высокая контрастность для читаемости при любом освещении.
- WCAG AA минимум, **AAA желательно** (контраст ≥ 7:1 для основного текста).
- Цвет — для семантики (ошибки красный, успех зелёный), не для декорации.
- Иконки — solid, не line, лучше читаемы издалека.

## Типографика

- **Минимум 18px** для основного текста.
- Заголовки: 24-48px.
- Шрифт — sans-serif, без декоративных deviation.
- Line-height ≥ 1.5.
- Контраст между уровнями заголовков — заметный (не 18px → 20px).

## Touch targets

- **Минимум 44×44px** для любого касаемого элемента (Apple HIG / Material).
- **Желательно 60×60px** для primary CTA на kiosk.
- Расстояние между кнопками ≥ 8px.
- Не делать touch-зоны вплотную к краям экрана (сложно попасть).

## Состояния и обратная связь

- Каждое нажатие → визуальный feedback ≤ 100ms.
- Долгие операции (>500ms) → progress indicator.
- Долгие операции (>3s) → счётчик / процент / "это займёт N секунд".
- Ошибки — конкретные, на русском, с путём решения.
- Успех — короткое подтверждение, авто-исчезающее.

## Accessibility

- ARIA labels на все touch elements.
- `aria-live` для уведомлений.
- `role` для кастомных элементов.
- Focus management при навигации.
- Поддержка screen reader (хотя в kiosk без экрана, но в админке важно).

## Компоненты (стиль Cal.com refinements)

- Минимум визуального шума.
- Чёткие линии разделения секций.
- Нет градиентов, теней — только сплошные заливки.
- Микроанимации только функциональные (transition между состояниями).

## Анимации

- Длительность: 150-250ms для feedback, 300-400ms для transitions.
- Easing: `ease-out` для входящих, `ease-in` для исчезающих.
- Отключаемо через `prefers-reduced-motion`.

## Информационная плотность

- Kiosk: ОДНО действие на экран. Большие кнопки.
- Не пытаться показать всю информацию сразу. Прогрессивно раскрывать.
- Хлебные крошки или прогресс-индикатор на multi-step flows.

## Мульти-язык

- Сейчас RU only.
- Архитектурно готовить к i18n (тексты в отдельных файлах, не хардкод в JSX).
- Готовиться к EN как минимум для production (Q1 2027).

## Brand / тон

- Дружелюбно, но кратко.
- Избегать жаргона ("RFID", "ИРБИС", "ЕКП" — допустимы только в админке).
- На kiosk: "приложите карту", "положите книгу", "готово".

## Проверочный чек-лист перед PR

- [ ] Контраст AA проверен (axe DevTools или Lighthouse).
- [ ] Все touch targets ≥ 44px.
- [ ] Шрифты ≥ 18px.
- [ ] Состояния hover/active/disabled видимы.
- [ ] Текст не упирается в края (padding ≥ 16px).
- [ ] Проверено на реальном экране kiosk (если меняется UI).
- [ ] ARIA labels на интерактивных элементах.
- [ ] Reduced-motion поддержан.
